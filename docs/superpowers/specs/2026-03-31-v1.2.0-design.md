# Onju Voice v1.2.0 — Design Spec

## Overview

Three new features for the Onju Voice ESPHome firmware, building on the v1.1.0 foundation. All features are HA-configurable and respect the shared I2S bus constraint.

**Target:** v1.2.0
**Base:** onju-voice.yaml (553 lines, v1.1.0)

---

## Feature 1: Voice Timers

### Summary

Users set timers via voice ("set a 5 minute timer"). The device shows a LED countdown bar and plays a local RTTTL alarm tone when the timer finishes. Touch-to-dismiss stops the alarm.

### How It Works

ESPHome's `voice_assistant` component exposes native timer events. No HA timer helpers needed — the voice pipeline handles everything.

**Lifecycle:**

1. User says "set a 5 minute timer"
2. HA conversation agent processes the request
3. `on_timer_started` fires on device → LED countdown begins (orange proportional bar)
4. `on_timer_tick` fires every second → LED bar shrinks as time remaining decreases
5. `on_timer_finished` fires → RTTTL alarm plays + LED pulses red
6. User touches center → alarm stops, LED resets to idle state

**Multiple timers:** `on_timer_tick` provides a `timers` vector with all active timers. The LED displays the timer closest to finishing. Each timer fires `on_timer_finished` independently.

**Cancel:** `on_timer_cancelled` fires when user cancels via voice → LED resets, no alarm.

### LED Behavior

| State | Color | Effect | LEDs |
|-------|-------|--------|------|
| Timer running | Orange | Proportional bar (like volume) | Top 4 LEDs, shrinks with remaining time |
| Timer alarm | Red | Pulse | All 6 LEDs |

**Priority integration with `reset_led`:**
- Volume display still wins over timer (highest priority)
- Timer alarm wins over all other states except volume
- Timer countdown wins over wake word idle, but loses to voice pipeline and media playback

Updated priority cascade:
1. Volume display (WHITE)
2. Timer alarm (RED pulse)
3. Media playing (TEAL / party mode)
4. TTS announcing (GREEN)
5. Voice pipeline active (WHITE / BLUE)
6. Timer countdown (ORANGE bar)
7. Wake word idle (PURPLE)
8. Off

### RTTTL Component

New component added to the config:

```yaml
rtttl:
  id: buzzer
  speaker: <tbd — see Volume Click section for speaker routing>
```

**Alarm tone:** A repeating melody that plays until dismissed. The `on_timer_finished` trigger starts a script that loops the RTTTL tone. Center touch or HA `button.press` stops it.

**Alarm dismiss script:**
- Mode: `single`
- Stops RTTTL playback
- Clears alarm state
- Calls `reset_led`

### HA Entities

| Entity | Type | Default | Description |
|--------|------|---------|-------------|
| Timer LED | Switch | ON | Show/hide LED countdown bar during active timers |

### Globals

| Name | Type | Purpose |
|------|------|---------|
| `timer_alarm_active` | bool | Tracks whether alarm is currently sounding |
| `active_timer_count` | int | Number of running timers (for LED logic) |

## Feature 2: Volume Click

### Summary

A short RTTTL click sound plays when the user adjusts volume via touch. The click is immediate and non-queued — if audio is already playing, the click is skipped rather than delayed.

### Speaker Routing

This is the key technical decision for both volume click and timer alarm.

**Option A: RTTTL as third mixer source**
Add a third source speaker to the mixer:

```yaml
speaker:
  - platform: mixer
    id: onju_mixer
    output_speaker: i2s_speaker
    queue_mode: true
    source_speakers:
      - id: announcement_speaker
      - id: media_speaker
      - id: alert_speaker         # NEW — for RTTTL
```

Problem: `queue_mode: true` means RTTTL queues behind active audio. This is correct for timer alarms (they should play after TTS finishes), but wrong for volume clicks (must be immediate or skipped).

**Option B: RTTTL bypasses mixer, writes directly to i2s_speaker**

```yaml
rtttl:
  id: buzzer
  speaker: i2s_speaker    # Direct to hardware
```

Problem: May conflict with mixer output if both write to i2s_speaker simultaneously.

**Recommended approach: Option A with conditional logic**

Use the mixer (option A) for clean bus arbitration, but handle volume click in software:

```yaml
# In adjust_volume script:
- if:
    condition:
      - media_player.is_idle: onju_media_player
    then:
      - rtttl.play:
          rtttl: "click:d=32,o=7,b=900:c"
```

Volume click only plays when the media player is idle (no music, no TTS). When something is playing, the volume changes silently — the LED bar already provides visual feedback.

Timer alarm uses the same `alert_speaker` through the mixer. Since it fires `on_timer_finished` (after voice pipeline is done), queue_mode works correctly — the alarm plays as soon as the current audio finishes.

### RTTTL Configuration

```yaml
rtttl:
  id: buzzer
  speaker: alert_speaker
  gain: 0.6               # Lower than media, adjust to taste
```

### HA Entities

| Entity | Type | Default | Description |
|--------|------|---------|-------------|
| Volume Sound | Switch | ON | Enable/disable click sound on volume change |

### Behavior Matrix

| Media player state | Volume touch | Result |
|-------------------|--------------|--------|
| Idle | Touch | Click sound + LED bar |
| Playing music | Touch | Silent + LED bar |
| Announcing TTS | Touch | Silent + LED bar |

## Feature 3: Party Mode

### Summary

Animated LED effects during music playback. Replaces the current on/off "Music Playback Light" switch with a select entity offering multiple modes.

### LED Effects

All effects run only on the **top 4 LEDs** (same partition as current music teal). Left/right LEDs remain available for volume display.

| Mode | Description | Speed |
|------|-------------|-------|
| `off` | No LED during music | — |
| `solid` | Current teal flicker (existing behavior) | — |
| `rainbow` | HSV hue rotation across 4 LEDs | ~2s full cycle |
| `fire` | Random flicker in red/orange/yellow | ~50ms updates |
| `chase` | Single bright pixel moving across LEDs | ~150ms step |
| `strobe` | Flash in random colors | ~100ms on/off |
| `cycle` | Rotates through rainbow → fire → chase → strobe | 30s per mode |

### Implementation

**New select entity** replaces the "Music Playback Light" switch:

```yaml
select:
  - platform: template
    id: music_led_mode
    name: "Music LED Mode"
    optimistic: true
    restore_value: true
    options:
      - "off"
      - "solid"
      - "rainbow"
      - "fire"
      - "chase"
      - "strobe"
      - "cycle"
    initial_option: "solid"
```

**Cycle mode** uses a script:

```yaml
script:
  - id: party_cycle
    mode: restart
    then:
      - while:
          condition:
            lambda: 'return id(music_led_mode).state == "cycle";'
          then:
            - light.turn_on: { id: top_led, effect: "rainbow" }
            - delay: 30s
            - light.turn_on: { id: top_led, effect: "fire" }
            - delay: 30s
            - light.turn_on: { id: top_led, effect: "chase" }
            - delay: 30s
            - light.turn_on: { id: top_led, effect: "strobe" }
            - delay: 30s
```

### Integration with reset_led

The `reset_led` script currently checks a boolean `music_light` switch. This changes to:

```
# In the "music playing" branch of reset_led:
if music_led_mode != "off":
    apply selected effect (or start cycle script)
```

### Migration

- Remove: `music_playback_light` switch
- Add: `music_led_mode` select
- Update: `reset_led` script to read select state instead of switch state

### HA Entities

| Entity | Type | Default | Description |
|--------|------|---------|-------------|
| Music LED Mode | Select | solid | LED behavior during music playback |

Removes: "Music Playback Light" switch (replaced by select)

## Updated LED Priority Cascade

```
1. Volume display         → WHITE proportional bar
2. Timer alarm            → RED pulse
3. Music playing          → per music_led_mode select
4. TTS announcing         → GREEN flicker
5. Voice recording        → WHITE twinkle
6. Voice processing       → BLUE scan
7. Timer countdown        → ORANGE proportional bar
8. Wake word idle         → PURPLE twinkle
9. Off
```

## Test Plan

### New automated tests needed

- Timer LED states: countdown bar, alarm, dismiss, multiple timers
- Timer priority: timer vs volume, timer vs music, timer vs voice
- Volume click: plays when idle, skipped when playing
- Party mode: each effect activates during music, stops when music stops
- Party cycle: rotates correctly, stops on mode change
- Music LED Mode select: all options produce correct LED state
- Migration: music_playback_light removal doesn't break existing states

### Manual verification

- Set timer via voice, watch LED countdown, verify alarm plays and touch-dismiss works
- Adjust volume during silence (hear click) and during music (no click)
- Play music, cycle through all party modes in HA
- Verify cycle mode rotates every 30s
- Set timer during music playback — verify alarm plays after music (or interrupts?)

## Out of Scope

- Beat-reactive LED (V2 — blocked by ESPHome not exposing speaker buffer data)
- Intercom
- Custom alarm tones from HA media library
- Timer display on external screen/display
